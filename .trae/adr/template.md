# ADR-NNNN: <一句话标题>

> 一句话标题用动词开头，描述"做了什么决策"，而不是"问题是什么"。

| 字段 | 值 |
|---|---|
| **状态** | proposed / accepted / superseded / deprecated |
| **日期** | YYYY-MM-DD |
| **作者** | <谁提议的> |
| **相关 ADR** | NNNN, NNNN |
| **Tags** | storage / llm / cli / api / infra / ux |

---

## 1. 背景（Context）

> 描述**决策时的处境**——不是问题本身，而是"为什么这是个值得记录的决策"。

- 当前项目处于什么阶段？
- 触发这次决策的事件是什么？（用户报告、PM 决策、技术债等）
- 有哪些**约束**？（时间、人力、QPS、平台兼容性等）
- 不做这个决策会怎样？

**示例**：
> 项目在 v0.3 时支持 3 个 CLI 命令，所有状态都靠文件系统手工管理。
> 准备上 v0.4（多项目并行），发现不同项目间会互相覆盖状态。
> 必须在 v0.4 release 前引入持久化。

---

## 2. 备选方案（Considered Options）

> 列出**至少 2 个**备选。不写"备选"等于没思考过 trade-off。

### 方案 A: <名称>
- **优点**: ...
- **缺点**: ...
- **成本**: 人力 X 天 / 引入新依赖 Y

### 方案 B: <名称>
- **优点**: ...
- **缺点**: ...
- **成本**: ...

### 方案 C: <名称>（如有）
- ...

---

## 3. 决定（Decision）

> 一句话陈述最终选择。

**采用方案 A：<名称>**

> 用粗体标注。决策要明确，不要"看情况"。

---

## 4. 原因（Rationale）

> 解释**为什么选这个**。这是 ADR 最有价值的部分——5 年后的人（或 AI）靠这段理解决策。

- 哪些约束是决定性的？
- 拒绝了 B/C 的**核心理由**是什么？
- 有什么**隐含假设**？如果假设不成立，何时需要回头重审？

---

## 5. 影响（Consequences）

### 正面
- 解决了 X 问题
- 打开了 Y 可能性

### 负面
- 引入了新依赖 / 复杂度
- 限制：未来想做 Z 时会有摩擦（写明！）

### 后续行动
- [ ] 添加文档到 docs/
- [ ] 添加测试覆盖
- [ ] 更新 memory.json（<哪些字段>）
- [ ] 通知团队（如果是多人项目）

---

## 6. 验证标准（Validation）

> 写明"什么情况下这个决策需要回头重审"——避免决策永远不更新。

- 触发条件 1：比如"QPS > 100 时需要重新评估"
- 触发条件 2：比如"团队 > 10 人时考虑升级到方案 B"
- 触发条件 3：比如"新版本 Bun 修复了方案 A 的某个 bug"
